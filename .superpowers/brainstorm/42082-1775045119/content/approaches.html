Three Approaches for HITL Relay v2

Pick the architecture that fits best. Each builds on the current relay differently.

A

Embedded — Everything in the Relay

Add SQLite, review UI, LLM-as-judge, and evaluation all inside src/index.ts. Single process, single deploy.

Pros

  • Simplest deploy — one process, one file (grows to ~3000 lines)
  • Zero latency between relay events and evaluation
  • No new dependencies beyond better-sqlite3

Cons

  • Monolith grows unwieldy — relay + eval + UI + LLM judge all coupled
  • LLM-as-judge calls block the event loop (unless worker thread)
  • Hard to test evaluation logic independently

Effort: ~2 weeks. Risk: Architect demon territory — one big file doing everything.

B

Modular Monolith — Split Domains, Single Process

Break into domain modules (src/relay/, src/eval/, src/ui/, src/store/) sharing one SQLite database. Still one npm start.

Pros

  • Clean domain boundaries (DDD) — relay, evaluation, storage, UI are separate
  • Each module testable independently
  • Single deploy, single process — same ops simplicity as A
  • LLM-as-judge runs in eval module with its own retry logic

Cons

  • Needs upfront file reorganization (~1 day)
  • Shared SQLite means modules are still coupled at the data layer

Effort: ~2.5 weeks. Risk: Low — standard Node.js project structure.

C

Sidecar — Relay + Eval as Separate Services

Keep relay lean (messaging only). Evaluation service runs alongside, subscribes to relay events via SSE (/observe), stores to its own SQLite, serves its own review UI.

Pros

  • Relay stays focused and lightweight — publishable as a clean OSS tool
  • Eval service can be swapped, upgraded, or disabled independently
  • Different teams/labs can run different eval configs
  • Eval service failure doesn't affect message delivery

Cons

  • Two processes to manage (needs launchd/pm2)
  • Network hop between relay and eval (minimal on localhost)
  • More complex setup for new users

Effort: ~3 weeks. Risk: Over-engineering if eval is only used by you.

⭐ Recommendation: B — Modular Monolith

B gives you clean DDD boundaries without ops overhead. The relay repo becomes a well-structured project with clear modules that lab participants can understand. You ship one npm start, not two services. If you later want to extract eval into a sidecar (approach C), the module boundaries make that a clean cut.

A is the architect demon. C is premature for 2-5 users.
